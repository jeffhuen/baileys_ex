# Documentation System

> **For Claude:** This is the single source of truth for the documentation strategy.
> Before writing, updating, or reviewing any documentation — read this file.
>
> **Companion files:**
> - [`user_docs/README.md`](../../user_docs/README.md) — user doc standards, templates, anti-patterns
> - [`user_docs/glossary.md`](../../user_docs/glossary.md) — canonical term definitions

---

## The Problem This Solves

Documentation fails in two ways:

1. **Drift** — docs describe what the code used to do. No one notices until a user reports it.
2. **Duplication** — the same fact lives in three places, maintained manually, guaranteed to diverge.

The solution: make the code the source of truth wherever possible, and have exactly one
canonical location for every fact that cannot be generated.

---

## Three Layers, Three Audiences

Every piece of documentation belongs to exactly one layer. Content never crosses layers.

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 1: Code Reference (public)                               │
│  Question: "How do I call this function / what does it do?"     │
│  Audience: Developers using BaileysEx as a dependency           │
│  Owner: @moduledoc / @doc / @spec / typedoc                    │
│  Generated by: mix docs                                         │
│  Output: doc/ (published to HexDocs)                            │
├─────────────────────────────────────────────────────────────────┤
│  Layer 2: User Docs (public)                                    │
│  Question: "How do I connect / send a message / handle groups?" │
│  Audience: Developers integrating BaileysEx into their app      │
│  Owner: user_docs/ — guides, reference, troubleshooting         │
│  Standards: user_docs/README.md (templates, anti-patterns)      │
│  Lifecycle: Hand-written, updated when APIs change              │
│  Output: Included in doc/ via mix.exs extras:                   │
├─────────────────────────────────────────────────────────────────┤
│  Layer 3: Architecture & Implementation (internal)              │
│  Question: "Why was this built this way / what's the plan?"     │
│  Audience: Contributors and agents working on BaileysEx         │
│  Owner: dev/implementation_plan/, dev/reference/                │
│  Lifecycle: Living during implementation, historical after      │
│  Output: NOT public — internal only                             │
└─────────────────────────────────────────────────────────────────┘
```

**One public output:** `mix docs` produces a single unified site (`doc/`) containing
Layers 1 and 2. Layer 3 stays internal.

## Layer 1: Code Reference

### What it owns

Every public module, function, type, and callback in `lib/baileys_ex/`.

- `@moduledoc` — module purpose, when to use it, key concepts
- `@doc` — per-function description, params, return values, examples
- `@spec` — accurate type signatures on all public functions
- `@typedoc` — type meaning where not self-evident

### Quality standard

A `@moduledoc` that merely restates the function signatures adds zero value. It must
tell the reader something they could not infer from the code alone: the module's role
in the system, when to use it vs alternatives, key concepts, and gotchas.

### What NOT to include

`@moduledoc`, `@doc`, and code comments must **never** reference internal artifacts:

| Bad | Good |
|-----|------|
| "Implements Phase 5 of the plan" | *(omit)* |
| "See GAP-44 for design rationale" | Explain the rationale inline if relevant |
| "Phase 8 will add retry logic" | "Retry logic will be added in a future release" |
| "Task 3.6a stub type constants" | *(omit — the code is self-describing)* |

Internal plan references are meaningless to ExDoc readers. Code comments follow the
same rule: `# TODO: add retry` is fine; `# TODO (Task 8.5):` is not.

## Layer 2: User Docs

### What it owns

Everything a developer needs to integrate BaileysEx into their application.

**Full standards, templates, and anti-patterns:** [`user_docs/README.md`](../../user_docs/README.md)

**Structure:**
```
user_docs/
├── README.md                          ← standards (not published)
├── glossary.md                        ← canonical term definitions
├── getting-started/
│   ├── installation.md                ← deps, compilation, Rust toolchain
│   ├── first-connection.md            ← QR pairing, phone pairing
│   └── sending-your-first-message.md  ← text message end-to-end
├── guides/
│   ├── messages.md                    ← all message types, reactions, polls
│   ├── media.md                       ← images, video, audio, documents
│   ├── groups.md                      ← group CRUD, participants, invites
│   ├── presence.md                    ← online/offline/composing status
│   ├── events-and-subscriptions.md    ← event subscription, handling patterns
│   ├── authentication-and-persistence.md
│   │                                   ← auth state, persistence, reconnection
│   ├── advanced-features.md           ← newsletters, communities, business, WAM
│   └── manage-app-state-sync.md       ← direct Syncd resync and patch workflows
├── reference/
│   ├── configuration.md               ← connection options, all config keys
│   ├── event-catalog.md               ← complete event type catalog
│   └── message-types.md               ← all supported message type structs
└── troubleshooting/
    ├── connection-issues.md           ← WebSocket, Noise handshake, timeouts
    ├── authentication-issues.md       ← QR failures, pairing code issues
    ├── encryption-issues.md           ← Signal session errors, key mismatches
    └── app-state-sync-issues.md       ← Syncd verification and key-state failures
```

**Examples:**
```
examples/
├── echo_bot.exs          — minimal working bot
└── echo-bot.md           — published example guide linking to the runnable script
```

**Included in `doc/` output via `mix.exs`:**
```elixir
docs: [
  extras: [
    "user_docs/glossary.md": [title: "Glossary"],
    "user_docs/getting-started/installation.md": [title: "Installation"],
    "user_docs/getting-started/first-connection.md": [title: "First Connection"],
    "examples/echo-bot.md": [title: "Echo Bot Example"]
  ],
  groups_for_extras: [
    "Getting Started": ~r/user_docs\/getting-started\//,
    "Guides": ~r/user_docs\/guides\//,
    "Reference": ~r/user_docs\/reference\//,
    "Troubleshooting": ~r/user_docs\/troubleshooting\//,
    "Examples": ~r/examples\//
  ]
]
```

### Rules

- All standards, templates, and anti-patterns are in [`user_docs/README.md`](../../user_docs/README.md)
- Guides are task-oriented ("how do I send an image") not API-oriented
- Every code example must be runnable or clearly marked as pseudocode
- If a guide references a function, link to the ExDoc page — don't duplicate the docs
- User docs pages are written in Phase 12 (Polish) once APIs are stable
- WhatsApp-specific terms must be defined in `user_docs/glossary.md` before first use

## Layer 3: Architecture & Implementation (Internal)

### What it owns

Internal documentation for contributors and agents building BaileysEx.

```
dev/
├── implementation_plan/
│   ├── 00-overview.md … 12-polish.md   — phase specs
│   ├── CLAUDE.md                        — agent workflow rules
│   ├── PROGRESS.md                      — task tracker
│   └── documentation-system.md          — this file
└── reference/
    ├── Baileys-master/                  — TypeScript reference source
    └── whatsapp-rust-main/              — Rust reference source
```

Not published. Not referenced from Layer 1 or Layer 2.

## Drift Prevention

| Risk | Prevention |
|------|-----------|
| `@moduledoc` contradicts actual behaviour | Delivery gate: `mix docs` must succeed without warnings |
| User docs describe unimplemented features | Anti-Pattern 6: if not implemented, it doesn't exist in docs |
| Same concept explained in multiple pages | One canonical home per concept, others link to it |
| Layer 1 references internal artifacts | No-internal-references rule (see Layer 1 above) |
| Public API changes without doc update | `@doc` lives next to the code — changed together or not at all |
| Glossary term redefined in a guide | `user_docs/glossary.md` is canonical — guides link, never redefine |
| User doc page not in index | `user_docs/README.md` Page Index must list every page |
| Feature ships without user guide | Delivery gate: answer 3 questions (setup? workflow? failure mode?) |

## What Runs, What You Get

```bash
mix docs
```

Output `doc/` contains:
- ExDoc API reference (`@moduledoc` / `@doc` / `@spec`)
- User docs from `user_docs/` (getting-started, guides, reference, troubleshooting, glossary)
- Examples
- All searchable, all navigable, published to HexDocs
